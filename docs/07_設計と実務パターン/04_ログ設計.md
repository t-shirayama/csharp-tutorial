# ログ設計と構造化ログ

## 目的

障害調査と運用に役立つログの設計観点と、`ILogger<T>` を使った構造化ログの書き方を理解します。

## 要点

- ログは後から事実を追うための記録です。
- レベル、イベントID、相関ID、構造化ログを意識します。
- 個人情報や秘密情報を出力しないようにします。
- ログは文字列ではなく、名前付きプロパティを持つイベントとして出します。
- 例外ログでは例外オブジェクトを `LogError` に渡します。
- CorrelationId や業務 ID を入れると、障害調査で追跡しやすくなります。

## コード例

```csharp
// この例では、ILogger<T> を DI で受け取り、構造化ログを出します。
public class OrderService
{
    private readonly ILogger<OrderService> logger;

    public OrderService(ILogger<OrderService> logger)
    {
        this.logger = logger;
    }

    public void Complete(int orderId)
    {
        logger.LogInformation("Completing order. OrderId={OrderId}", orderId);

        try
        {
            // 業務処理
            logger.LogInformation("Order completed. OrderId={OrderId}", orderId);
        }
        catch (Exception exception)
        {
            logger.LogError(exception, "Failed to complete order. OrderId={OrderId}", orderId);
            throw;
        }
    }
}
```

## 悪い例

```csharp
logger.LogInformation($"Order completed. OrderId={orderId}");
```

文字列補間では、ログ基盤が `OrderId` を構造化された値として扱えません。

## CorrelationId の入口

```csharp
using (logger.BeginScope(new Dictionary<string, object>
{
    ["CorrelationId"] = correlationId
}))
{
    logger.LogInformation("Processing request.");
}
```

## コードの読み方

`ILogger<OrderService>` は DI で注入される logger です。`"OrderId={OrderId}"` のような placeholder を使うと、ログ基盤が `OrderId` を検索可能な値として扱えます。

`LogError(exception, ...)` の第1引数に例外オブジェクトを渡すと、stack trace を含めて記録できます。`BeginScope` を使うと、同じ処理範囲のログに共通の値を付けられます。

## 実務での使い方

API リクエスト、外部サービス呼び出し、バッチ処理、重要な業務イベントで使います。ログは「何が起きたか」「どの対象か」「なぜ失敗したか」を後から追える粒度にします。

リクエスト単位の相関ID、業務ID、外部連携先、失敗理由を追えるようにします。ログレベルは Debug / Information / Warning / Error を使い分けます。

## よくあるミス

- 文字列連結で構造化情報を失う。
- Error ログに例外オブジェクトを渡さない。
- 成功ログを出しすぎて重要なログが埋もれる。
- パスワード、トークン、個人情報を出す。

## レビュー観点

- 文字列補間ではなく、構造化ログのプレースホルダーを使っているか。
- Error ログには例外オブジェクトが渡されているか。
- ユーザー ID、注文 ID、CorrelationId など調査に必要なキーがあるか。
- PII や秘密情報がログに出ない設計になっているか。
- 例外ログが1か所に集約されているか。
- ログだけで業務上の状態遷移を追えるか。
