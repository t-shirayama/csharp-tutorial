# プロジェクト作成と実行

## 目的

`dotnet` CLI でプロジェクトを作り、restore、build、run、test の流れを確認します。

## 要点

- 注意: repository root ではなく別フォルダで `dotnet build` している。
- 注意: test project を solution に追加し忘れる。
- 注意: `dotnet build` の失敗を compiler error だと思ったが、実際は restore で失敗している。
- 注意: `bin` や `obj` を編集・コミットしようとする。

## コンソールアプリを作る

```powershell
# HelloCsharp という名前の console project を作成します。
dotnet new console -n HelloCsharp

# 作成された project folder に移動します。
Set-Location HelloCsharp

# project が必要とする NuGet package と依存関係情報を復元します。
dotnet restore

# source code を compile して、エラーがないか確認します。
dotnet build

# console app を実行します。
dotnet run
```

`dotnet run` で `Hello, World!` が表示されれば成功です。

## `dotnet restore` の意味

`dotnet restore` は、project file に書かれた NuGet package を取得し、build に必要な依存関係情報を `obj` 配下へ作るコマンドです。source code を compile するコマンドではありません。

`dotnet build`、`dotnet run`、`dotnet test` は必要に応じて restore を自動実行します。ただし、実務や CI では `dotnet restore`、`dotnet build --no-restore`、`dotnet test --no-build` のように分けると、package 復元、compile、test のどこで失敗したかを切り分けやすくなります。

restore が失敗した場合は、SDK の問題だけでなく、NuGet source、社内 feed の認証、package version、ネットワーク制限を確認します。

## solution を作る

複数プロジェクトを扱う場合は solution を使います。

solution は、複数の project をまとめて扱うための入れ物です。`.csproj` はアプリ本体、ライブラリ、テストなど個別の project を表し、solution file はそれらをまとめて build / test できるようにします。

.NET 10 SDK の `dotnet new sln` は既定で `.slnx` を作ります。.NET 9 以前では `.sln` が作られます。どちらの場合も、ここで扱う目的は「複数 project をまとめること」です。

実務では `src` にアプリ本体やライブラリ、`tests` にテスト project を置く構成がよく使われます。solution root で `dotnet build` や `dotnet test` を実行できるようにしておくと、VS Code、CI、レビュー時の確認手順を揃えやすくなります。

```powershell
# SampleApp という solution file を作成します。
dotnet new sln -n SampleApp

# アプリ本体の処理を置く class library project を src 配下に作成します。
dotnet new classlib -n SampleApp.Core -o src/SampleApp.Core

# テスト用の xUnit project を tests 配下に作成します。
dotnet new xunit -n SampleApp.Core.Tests -o tests/SampleApp.Core.Tests

# 本体 project を solution に登録します。
dotnet sln add src/SampleApp.Core/SampleApp.Core.csproj

# テスト project を solution に登録します。
dotnet sln add tests/SampleApp.Core.Tests/SampleApp.Core.Tests.csproj

# テスト project から本体 project の code を参照できるようにします。
dotnet add tests/SampleApp.Core.Tests/SampleApp.Core.Tests.csproj reference src/SampleApp.Core/SampleApp.Core.csproj

# solution root からテストを実行し、登録済み project をまとめて確認します。
dotnet test
```

`dotnet sln add` は project を solution に登録するコマンドです。`dotnet add ... reference ...` は、テスト project から本体 project を参照するためのコマンドです。この2つは役割が違うため、どちらか一方だけでは `dotnet test` で期待通りに動かないことがあります。

## 既存リポジトリを確認する

```powershell
# sample repository を local に取得します。
git clone https://github.com/example/sample-app.git

# 取得した repository folder に移動します。
Set-Location sample-app

# clone 直後に未追跡・変更済み file がないか確認します。
git status

# project が必要とする NuGet package と依存関係情報を復元します。
dotnet restore

# compile できるか確認します。
dotnet build

# test が通るか確認します。
dotnet test
```

clone 直後に build / test が通るか確認すると、自分の変更で壊したのか、元から環境差分があるのかを分けやすくなります。

## NuGet source を確認する

restore が失敗するときは、package source を確認します。

```powershell
# 現在有効な NuGet package source を一覧表示します。
dotnet nuget list source
```

社内 feed や token が関係する場合、認証情報をチャットや issue に貼らないよう注意します。
