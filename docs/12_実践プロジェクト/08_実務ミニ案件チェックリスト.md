# 実務ミニ案件チェックリスト

## 目的

学習用アプリを、実務の小さな案件として進める順序、PR 提出前の確認、完成度を上げる観点を整理します。

## 作るもの

TODO、家計簿、メモ API などを題材に、要件整理から実装、品質確認、PR 提出までを一通り行います。

## 学習ポイント

- 要件を小さな issue に分ける
- API、DB、テスト、ログ、設定を段階的に追加する
- README に実行手順と確認観点を残す
- PR とレビューを想定して変更を説明する
- 正常系だけでなく異常系、境界値、運用時の調査観点を見る
- 仕様変更、破壊的変更、未対応事項を隠さない

## 手順

1. 作る機能を3つ以内に絞る。
2. 入力、出力、エラー、保存先を決める。
3. 最小実装を作る。
4. テストを追加する。
5. ログ、設定、例外処理を整える。
6. README と確認コマンドを更新する。
7. PR 提出チェックを行う。

## 完成度チェックリスト

- 入力 validation がある。
- エラー時の response や message が分かりやすい。
- 重要な処理に structured log がある。
- secret が repository とログに出ていない。
- 設定値が環境ごとに切り替えられる。
- unit test と必要な integration test がある。
- `dotnet format` または同等の整形を実行した。
- README に起動、テスト、設定、トラブルシュートがある。

## PR 提出チェックリスト

- 何を、なぜ変更したかを書いた。
- `dotnet build` を実行した。
- `dotnet test` を実行した。
- 追加・変更した仕様にテストがある。
- 設定値、secret、migration の影響を確認した。
- API 変更がある場合、OpenAPI と README を更新した。
- ログに個人情報や secret が出ないことを確認した。

## PR 本文例

```markdown
## Summary

- TODO 作成 API を追加
- 入力 validation と ProblemDetails response を追加

## Checks

- dotnet build
- dotnet test

## Notes

- DB migration あり
```

## 発展課題

- CI で build/test/format を実行する。
- Docker で DB などの依存サービスを起動する。
- OpenAPI とサンプル request を整える。
- PR template を作る。
- review 指摘を分類して再発防止メモにする。
- release note を PR から作る。

## 完了条件

- `dotnet build` と `dotnet test` が通る。
- README の手順で動作確認できる。
- 主要な正常系、異常系、境界値のテストがある。
- ログとエラーレスポンスが調査に使える。
- レビュー担当者が変更内容と確認方法をすぐ把握できる。
- 初見の人が README だけで起動できる。
