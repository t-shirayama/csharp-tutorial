# 楽観的同時実行制御

## 目的

複数ユーザーが同じデータを更新するときの競合を検出し、安全に扱えるようにします。

## 前提

- [DbContext と Entity](04_DbContextとEntity.md) を読んでいる
- [Transaction](06_Transaction.md) を読んでいる

## 要点

- 楽観的同時実行制御は「更新時に競合を検出する」方式です。
- EF Core では concurrency token を使います。
- 競合時は自動で上書きせず、再読み込みやユーザー確認を検討します。

- 注意: 後から保存した人が常に勝つ仕様にしてしまう。
- 注意: 競合を 500 として扱う。
- 注意: UI や API 利用者に再試行方法を伝えない。
- 注意: concurrency token を response/request に含める設計を忘れる。

## row version の例

```csharp
// この例では「楽観的同時実行制御」の要点を最小のコードで確認します。
public class TodoItem
{
    public int Id { get; set; }
    public string Title { get; set; } = "";

    [Timestamp]
    public byte[] RowVersion { get; set; } = [];
}
```

## 実務での使い方

在庫、予約、承認、金額変更など、後勝ち上書きが危険なデータで使います。API では競合を 409 Conflict として返す設計がよくあります。
